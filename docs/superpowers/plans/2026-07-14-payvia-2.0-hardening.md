# Payvia 2.0.0 Hardening Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship Payvia 2.0.0 — integer minor units everywhere + tenant-sentinel tenancy — per `docs/superpowers/specs/2026-07-14-payvia-2.0-hardening-design.md` (revision 4, fold posture).

**Architecture:** Payvia has git tags but ZERO deployed databases (owner-confirmed), so ALL schema changes fold into the original create-table migrations — no upgrade migrations, no data conversion, no preflight/rebuild machinery. Migration 007 (a released-style rebuild) folds into 002 and is deleted. The runtime becomes pure integer passthrough (gateways stop dividing, `PaymentService::minorUnits()` is deleted); tenancy mirrors commerce's sentinel module with an explicit interactive-vs-correlation repository split and fail-closed webhook ownership.

**Tech Stack:** PHP 8.3, Glueful framework ^1.57 (path-dev), PHPUnit (101-test baseline), extension-contracts path dep.

## Global Constraints

- Repos: `~/Sites/glueful/extensions/payvia` (Tasks 1–7) and `~/Sites/glueful/extensions/subscriptions` (Task 8 only). Branch `dev` in both. Commits at group boundaries, conventional style, NO attribution trailers. Never stage `docs/superpowers/**` or `.superpowers/**`.
- **FOLD POSTURE (owner-confirmed: tags exist, nothing deployed):** every schema change edits the ORIGINAL create-table migrations to their final shape. NO upgrade migrations, NO data-conversion SQL, NO preflight, NO shadow rebuilds. Migration 007 (the billing_plans unique rescope rebuild) folds into 002 and is deleted; 008 renames to 007 to close the gap (file + class). Fresh installs run the clean sequence; any pre-2.0 dev DB is synced/recreated manually (owner's standing rule).
- Raw provider payloads (`provider_events.raw_payload`, `payments.raw_payload`) are immutable evidence — runtime never rewrites them. Newly normalized events carrying a numeric amount also carry `amount_unit: 'minor'` (forward-compat marker).
- Zero float money arithmetic anywhere after Task 2: gateways return `int`/`?int`, `PaymentService::minorUnits()` is REMOVED (grep-gate its absence), storage/`PayableReference`/`PaymentConfirmation`/API/events all carry the same integer.
- Tenancy: sentinel `tenant_uuid` char(12) default `''`; interactive repository methods resolve tenant through the Payvia-local `PayviaTenantResolver` service and constrain tenant+identity with non-revealing not-found. Payvia NEVER binds/replaces shared `CurrentTenantResolver`. Tenantless webhook correlation is isolated in one internal repository and, when tenancy is active, runs its query-builder work through the neutral `TenantContextRunner::runAsSystem()` contract. This preserves QueryExecutor/write-barrier interception while deliberately bypassing row scope; every mutation predicate still includes the located row's `tenant_uuid`. Global keys stay global: all `uuid`s, `payments.reference`, `(gateway, gateway_subscription_id)`.
- Route middleware: three ordered config profiles (`payvia.security.auth_middleware` default `['auth']`, `tenant_context_middleware` default `[]`, `manage_middleware` default `['admin']`); webhook route uses none. Payvia never names host middleware aliases.
- Single-store regression: with defaults (sentinel, empty tenant profile), the pre-existing 101-test suite behavior is preserved (amount assertions update ×100 — that's the feature; anything else changing is a red flag to surface).
- Quality gates per group: `composer test`, `composer phpcs`, and the verified `composer analyze` script — all clean.
- Version: bump `composer.json` `extra.glueful.version` to `2.0.0` in Task 7 and fix `composerVersion()` to read that real path (there is no top-level `version`). NO git tags (owner tags).
