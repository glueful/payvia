# Payvia 2.0.0 — Money & Tenancy Hardening

**Status:** revision 4 — fold posture (owner decision 2026-07-15: tags v0.7.0–v1.1.0 exist
but ZERO deployed databases, so the pre-launch folding rule governs; all schema changes edit
the original create-table migrations and the upgrade-migration machinery is discarded)
**Repo:** `glueful/payvia` only (extension-contracts unchanged — `PayableReference` already
mandates integer minor units; Payvia's own repository interfaces may break inside the major).
Consumers: `glueful/subscriptions` keeps Payvia as a soft suggestion and runs a separate
Payvia-present compatibility lane after release; Composer suggestions cannot be version-pinned.
A tenancy-enabled host must configure Payvia's neutral tenant-context middleware profile;
Payvia never names host-specific middleware aliases.
**Breaking:** yes — major release (API amount semantics change and v1.x tags exist as
identifiers). But with no deployed database anywhere, both changes FOLD into the original
create-table migrations: no upgrade migrations, no data conversion, no preflight/rebuild
machinery. Pre-2.0 dev installs recreate/sync manually. If a v1 deployment materializes
before 2.0 ships, this posture must be revisited.

## Why now

1. **Payvia violates its own contract boundary.** `PayableReference.amount` is documented
   "always integer minor units", and `payment_intents.amount` (008) stores it that way —
   but `payments`/`invoices`/`billing_plans` store `decimal(12,2)` majors, and the
   gateways *receive* minor units from Stripe/Paystack then lossily float-divide by
   `100.0` to store majors (PaystackGateway:91,323; StripeGateway:171-197,316). The
   codebase is split-brain across its own tables, with float money arithmetic in the
   middle.
2. **No tenancy.** No table carries `tenant_uuid`; `invoices.number` is globally unique.
   Any multi-workspace app (Thallo) bridging commerce↔payvia would leak receivables
   across workspaces.

## Part 1 — Integer minor units everywhere

**Target state:** every stored/transported amount in Payvia is an integer in the
currency's minor unit. Zero float arithmetic on money.

- **Schema (folded):** `payments.amount` (001), `billing_plans.amount` (002),
  `invoices.amount` (003) become `bigInteger` directly in their create-table migrations.
  There is no data to convert — no preflight, no shadow rebuild, no per-driver conversion
  SQL. Migration 007 (the billing_plans unique rescope rebuild) folds its final
  `(gateway, name)` unique shape into 002 and is deleted; 008 renumbers to 007.
  - `currency` columns: keep width, default stays `GHS`.
- **Runtime conversion is complete, not gateway-only:** delete every `/100.0` gateway
  conversion and every downstream float cast/re-conversion. Stripe/Paystack amounts pass
  through as integers (they are already minor units on the wire); gateway normalization
  methods return `int`/`?int`; `PaymentService` removes `minorUnits()` and sends the same
  integer to storage, `PayableReference`, `PaymentConfirmation`, API responses, and normalized
  events. Amount comparisons become integer equality. No runtime Money/exponent helper is
  needed because Payvia never formats money for display.
- **API surface (breaking):** create-invoice / billing-plan / list endpoints read and
  return integer minor units. DTO/validation: `integer` rule + positivity domain checks.
  CHANGELOG carries explicit before/after examples (`10.50` → `1050`).
- **Normalized events:** runtime-normalized events carrying a numeric amount also carry
  `amount_unit: 'minor'` (a forward-compat marker for any future consumer/re-normalizer).
  There are no historical events to rewrite. Raw provider payloads are immutable evidence
  and are never rewritten; `payments.raw_payload` is likewise opaque historical evidence
  rather than a typed amount contract.
- **Internal consistency check:** `payment_intents.amount` is already the target type, but its
  callers and comparisons are still swept for floats; invoice/billing-plan controllers,
  services, repositories, OpenAPI examples, fixtures, and tests all use integers.

## Part 2 — Tenant sentinel (commerce pattern)

Mirror the commerce extension's proven tenancy posture:

- **Columns (folded into the create migrations):** `tenant_uuid` char(12) default `''` +
  index on the domain tables: `payments` (001), `billing_plans` (002), `invoices` (003),
  `gateway_subscriptions` (005), `payment_intents` (007, renumbered). `provider_events`
  deliberately remains a global transport/inbox table: signed webhooks arrive without
  request tenant context, and each domain applier must resolve ownership before touching a
  tenant-bearing table.
- **Unique-key classification is explicit:**
  - globally unique correlation identities stay global: every table's `uuid`,
    `payments.reference`, and `(gateway, gateway_subscription_id)`. Tenantless callbacks and
    reconciliation depend on these identities being unambiguous;
  - tenant business keys become composite: `invoices (tenant_uuid, number)`,
    `billing_plans (tenant_uuid, gateway, name)`, and
    `payment_intents (tenant_uuid, idempotency_key)`;
  - these are the CREATE-TABLE shapes (folded) — the old global forms never exist in 2.0.
    The schema-shape test proves the composite semantics with two-tenant duplicate inserts.
- **Payment-intent semantics:** `findOpen/createOpen/close` resolve the tenant first and
  constrain tenant + payable/uuid. Two tenants may therefore open the same logical
  `{payable_type}:{payable_id}` without colliding; a close cannot reach another tenant's row.
- **Resolution:** a Payvia-local `PayviaTenantResolver` service consumes the shared
  `CurrentTenantResolver` when the host has one and wraps it fail-closed; otherwise it uses a
  sentinel resolver. Payvia never binds or replaces the shared contract. This mirrors
  commerce's sentinel/fail-closed/adopter behavior while preserving provider ownership.
  Single-store installs remain byte-identical (`''` everywhere).
- **Repository profiles, not one contradictory rule:**
  - interactive/application reads and writes always resolve the current tenant and constrain
    `tenant_uuid + identity`; cross-tenant identifiers return the same non-revealing not-found
    result as unknown identifiers;
  - one named internal `ProviderCorrelationRepository` may read by a globally unique provider
    key without request tenant context. When tenancy is active it executes query-builder work
    inside the neutral contracts-package `TenantContextRunner::runAsSystem()` seam; this
    deliberately suspends row scoping while preserving QueryExecutor interception, Thallo's
    mutation barrier, and query observability. With no tenancy resolver/runner, the same code
    executes normally for single-store mode. Resolver-present + runner-missing fails closed.
    The repository is not controller-visible, supports only exact provider-subscription and
    billing-plan UUID correlation needed today, and every mutation includes the located
    `tenant_uuid`, so correlation cannot transfer ownership. Runtime direct-PDO bypasses for
    these five tables are forbidden by a static test.
- **Subscription webhook ownership:** `GatewaySubscriptionService` follows this order under
  the global `(gateway, gateway_subscription_id)` identity:
  1. Existing projection: its persisted `tenant_uuid` is authoritative; update that exact
     tenant row and ignore/reject conflicting incoming ownership hints.
  2. First projection: derive the tenant from the globally unique `billing_plan_uuid` carried
     in normalized provider metadata and re-read that plan. If an explicit metadata
     `tenant_uuid` is also present, it must equal the plan owner. The signed webhook proves
     provider origin, but metadata alone is never an ownership authority.
  3. No existing projection and no valid local plan correlation: fail closed. Do not write a
     sentinel subscription row. Leave the provider event failed/retryable and diagnose the
     unresolved correlation.
  Reconciliation uses the same global locator and returns the existing owner; it never adopts
  or moves a projection between tenants.
- **Payment/provider events:** payment reference remains globally unique for future exact
  callback correlation, but Payvia 2.0 has no payment webhook applier: payment persistence is
  the authenticated, tenant-scoped `PaymentService::confirmAndRecord()` path. Generic
  `provider_events` remain tenantless and their dispatch contract is not presented as an
  ambient tenant context. A future payment applier must extend the audited correlation
  repository rather than opening an ad-hoc global query.

- **Operability:** Payvia ships `payvia:tenancy:adopt --tenant=<uuid>` over `TenantAdopter` and
  `payvia:diagnose` with resolver mode, registered tables, sentinel counts, and unresolved
  subscription-ownership failures. An unresolved owner is a typed failure recorded on the
  provider event, so retries and diagnostics agree on its classification.

### Tenant context on HTTP routes

The resolver reads request-scoped `TenantContext`; binding a `CurrentTenantResolver` alone does
not populate it. Payvia 2.0 therefore splits its route middleware configuration into three
ordered profiles:

1. `payvia.security.auth_middleware` (default `['auth']`),
2. `payvia.security.tenant_context_middleware` (default `[]` for byte-identical single-store
   installs),
3. `payvia.security.manage_middleware` (default `['admin']`).

Authenticated read/confirm routes use profile 1 → 2; management routes use 1 → 2 → 3. The
webhook route uses none of them and stays signature-authenticated/tenantless. A Thallo host
configures profile 2 as `tenant_profile:admin`, `tenant_bootstrap`,
`admin_tenant_binding`; Payvia itself never imports or hardcodes those aliases. When Payvia
tenancy is enabled and the profile failed to establish context, `FailClosedTenantResolver`
rejects the operation instead of falling back to `''`.

This is a 2.0 config break: a host that overrode the v1 `manage_middleware` list moves its
authentication entries into `auth_middleware` and leaves only authorization checks in
`manage_middleware`. The upgrade guide includes the exact before/after shape.

- **Tests:** two-tenant coverage per surface (invoices CRUD, payments lookup, plan
  uniqueness, same intent key in both tenants), trusted webhook correlation (existing owner,
  first projection through plan, mismatch, unresolved), route-profile ordering/fail-closed,
  and adoption/diagnostics parity with commerce.

## Sequencing

1. Fold money schema to final shape (001/002/003 bigInteger amounts; 007 folded into 002 and
   deleted; 008 renumbered) and the complete gateway/service/controller integer sweep.
2. Fold tenant columns + classified unique shapes into the create migrations, Tenancy module,
   neutral route profile, repository/correlation split, and two-tenant/webhook tests.
3. CHANGELOG (2.0.0 — schema finalized pre-deployment, no data-migration path; pre-2.0 dev
   installs recreate/sync manually), README API examples.
4. `glueful/subscriptions`: keep the normal Payvia-absent suite unchanged, then run a separate
   temporary Payvia-2.0-present bridge/reconcile lane. Payvia is only a Composer suggestion,
   so there is no version constraint to repin. Commit/release subscriptions only if that lane
   proves an adaptation or documentation change is necessary.

## Non-goals

- Refund support in Payvia (`RefundCollector` implementation) — separate feature, later.
- Commerce↔Payvia invoice bridging (pay-later checkout) — composition feature, later.
- Any display/formatting layer — Payvia has none and gains none.

## Regression gate

Full payvia suite green (101-test baseline plus new coverage) with:

- single-store behavior byte-identical under sentinel tenancy and empty tenant middleware;
- the schema-shape test proves a fresh migrate yields integer amount columns, the composite
  tenant business keys (with the old global forms absent — two-tenant duplicate inserts), the
  folded `(gateway, name)` plan unique, and intact global correlation keys;
- gateway fixtures assert integer pass-through and `PaymentService` never multiplies again;
- two tenants can reuse invoice numbers, plan names, and payment-intent keys while global
  provider references remain unambiguous;
- HTTP routes prove tenant middleware ordering and fail closed when context is absent;
- signed subscription webhooks cannot create ownerless/sentinel projections, transfer an
  existing projection, or accept a metadata tenant that disagrees with the local plan; and
- runtime direct-PDO bypass of the five tenant tables is absent; correlation proves it passes
  through `TenantContextRunner::runAsSystem()` and QueryExecutor.
